dockerCE-storage-overview

在Docker中管理数据

默认情况下，在容器内创建的所有文件都存储在可写的容器层上。这带来了几个影响：

容器内部创建的数据在容器被移除后不会持久化，这使得在其他进程需要访问数据时变得困难。
容器的可写层与运行容器的主机机器密切相关，限制了数据在其他环境中的可移植性。
向容器的可写层写入数据需要使用存储驱动程序来管理文件系统，这通过 Linux 内核实现的联合文件系统引入了额外的抽象层。这种抽象层可能会导致性能下降，相对于直接向主机文件系统写入数据卷而言。
Docker 提供了两种选项，允许容器将文件存储在主机机器上，确保即使容器停止后数据也会持久化：数据卷（volumes）和绑定挂载（bind mounts）。

此外，Docker 还支持容器将文件存储在主机机器的内存中，尽管这些文件不会持久化。在 Linux 系统上，Docker 使用 tmpfs 挂载将文件存储在主机的系统内存中，而在 Windows 系统上，则使用命名管道实现内存中的文件存储。

选择正确的挂载类型

无论您选择使用哪种挂载类型，从容器内部看，数据都是相同的。它在容器的文件系统中显示为一个目录或一个单独的文件。

将数据挂载为卷（Volumes）、绑定挂载（Bind Mounts）、或 tmpfs 挂载的区别可以通过想象数据在 Docker 主机上的位置来更容易理解。

挂载类型及其在 Docker 主机上的位置:

  • 卷（Volumes）存储在由 Docker 管理的主机文件系统的一部分中（在 Linux 上是 /var/lib/docker/volumes/）。非 Docker 进程不应该修改这部分文件系统。卷是在 Docker 中持久化数据的最佳方式。

  • 绑定挂载（Bind Mounts）可以存储在主机系统的任何位置。它们甚至可能是重要的系统文件或目录。Docker 主机上的非 Docker 进程或 Docker 容器可以随时修改它们。

  • tmpfs 挂载（tmpfs mounts）仅存储在主机系统的内存中，永远不会写入主机系统的文件系统。

绑定挂载和卷（Volumes）都可以使用 -v 或 --volume 标志挂载到容器中，但每种类型的语法略有不同。对于 tmpfs 挂载，您可以使用 --tmpfs 标志。我们建议在容器和服务中都使用 --mount 标志来挂载绑定挂载、卷（Volumes）或 tmpfs 挂载，因为这样的语法更加清晰易懂。

卷（Volumes）

卷（Volumes）由 Docker 创建和管理。您可以使用 docker volume create 命令显式创建卷，也可以在容器或服务创建过程中由 Docker 创建卷。

创建卷时，它存储在 Docker 主机上的一个目录中。当您将卷挂载到容器中时，实际挂载的就是这个目录。这与绑定挂载的工作方式类似，只不过卷是由 Docker 管理的，并且与主机机器的核心功能隔离。

一个给定的卷可以同时挂载到多个容器中。当没有运行的容器使用一个卷时，该卷仍然可供 Docker 使用，不会自动删除。您可以使用 docker volume prune 命令删除未使用的卷。

挂载卷时，它可能是具名或匿名的。匿名卷会被赋予一个随机名称，在给定的 Docker 主机中保证唯一性。与具名卷类似，匿名卷即使在移除使用它们的容器后仍然保留，除非您在创建容器时使用了 --rm 标志，此时匿名卷会被销毁。如果您连续创建多个使用匿名卷的容器，每个容器都会创建自己的卷。匿名卷不会自动被重用或在容器之间共享。要在两个或多个容器之间共享匿名卷，您必须使用随机卷 ID 挂载匿名卷。

卷还支持使用卷驱动程序（volume drivers），使您可以将数据存储在远程主机或云提供商等地方。

绑定挂载（Bind mounts）

Bind mounts（绑定挂载）与卷（Volumes）相比具有较为有限的功能。使用绑定挂载时，主机机器上的文件或目录会被挂载到容器中。这些文件或目录是通过它们在主机机器上的完整路径进行引用的。在 Docker 主机上，这些文件或目录不需要事先存在。如果它们尚不存在，则会根据需要创建。绑定挂载速度较快，但依赖于主机机器的文件系统具有特定的目录结构。如果您正在开发新的 Docker 应用程序，请考虑使用具名卷。您无法使用 Docker CLI 命令直接管理绑定挂载。
重要提示：

tmpfs

Tmpfs挂载（tmpfs mount）在Docker主机或容器内部都不会被持久化到磁盘上。它可以在容器的生命周期内使用，用于存储非持久状态或敏感信息。例如，在内部，Swarm服务使用tmpfs挂载将机密挂载到服务的容器中。

命名管道（Named pipes）

命名管道可以用于Docker主机与容器之间的通信。常见用例是在容器内部运行第三方工具，并使用命名管道连接到Docker Engine API。

卷的典型用例

卷是在Docker容器和服务中持久保存数据的首选方式。一些卷的典型用例包括：

  • 共享数据：
    在多个运行中的容器之间共享数据。如果未显式创建卷，那么当卷第一次被挂载到容器中时会自动创建。即使容器停止或被移除，卷仍然存在。
    多个容器可以同时挂载同一卷，无论是读写还是只读方式。卷只有在明确删除时才会被移除。

  • Docker主机无法保证特定目录或文件结构：
    卷有助于将Docker主机的配置与容器运行时分离，提供灵活性和抽象性。

  • 将容器数据存储在远程主机或云服务提供商上：
    卷使您可以将容器数据存储在远程主机或云平台上，而不是在本地。

  • 数据备份、恢复或迁移：
    卷提供了一种方便的方式来管理数据备份、恢复或在不同Docker主机之间迁移。
    使用卷的容器可以停止，然后备份卷的目录（例如/var/lib/docker/volumes/）。

  • 在Docker Desktop上进行高性能I/O操作：
    对于需要在Docker Desktop上进行高速I/O操作的应用程序，卷是首选。卷存储在Linux虚拟机中而不是主机上，从而使读写操作具有更低的延迟和更高的吞吐量。

  • 在Docker Desktop上需要完全本机文件系统行为：
    某些应用程序，例如需要对磁盘操作进行精确控制以保证事务持久性的数据库引擎，受益于使用卷。
    卷存储在Linux虚拟机中，并且可以提供这些保证，而绑定挂载是远程到macOS或Windows，在这些文件系统上的行为略有不同。

卷提供了在Docker环境中管理持久数据的多功能和高效方式，具有数据共享、可移植性、备份、性能和本机文件系统支持方面的优势。

绑定挂载的典型用例

通常情况下，尽可能使用卷。绑定挂载适用于以下类型的用例：

从主机机器共享配置文件到容器：
这是Docker默认提供DNS解析到容器的方式，通过将主机机器上的/etc/resolv.conf挂载到每个容器中实现。

在Docker主机上的开发环境与容器之间共享源代码或构建产物：
例如，您可以将Maven的target/目录挂载到容器中，在Docker主机上构建Maven项目时，容器就能获得重新构建后的产物。

在开发中，当Docker主机的文件或目录结构可以保证与容器所需的绑定挂载一致时，您生产的Dockerfile应直接将生产就绪的产物复制到镜像中，而不是依赖于绑定挂载。
综上所述，绑定挂载适用于需要在Docker主机和容器之间共享文件或目录，并且具有固定的结构和依赖关系的情况。

tmpfs 挂载典型用例

tmpfs挂载最适合的情况是，当您不希望数据在主机机器或容器内持久化时。这可能出于安全原因，或者是为了保护容器的性能，特别是在您的应用程序需要写入大量非持久状态数据时。

使用绑定挂载或卷的提示

  • 当你将一个空的卷挂载到容器中已存在文件或目录的目录中时，这些文件或目录会被复制到卷中。同样地，如果你启动一个容器并指定一个不存在的卷，系统会为你创建一个空的卷。这是一种很好的方式来预先填充另一个容器需要的数据。

  • 如果你将一个绑定挂载或非空卷挂载到容器中已存在文件或目录的目录中，这些文件或目录会被挂载所覆盖，就像在Linux主机上将文件保存到/mnt，然后将USB驱动器挂载到/mnt一样。直到USB驱动器被卸载，/mnt的内容会被USB驱动器的内容所遮蔽。被遮蔽的文件不会被删除或修改，但在绑定挂载或卷挂载时是不可访问的。
